# Test suite perf baseline

Wall-clock time for the full bats suite at each `--jobs` setting on
the project's devcontainer (`nproc` = 8). All runs exercise every
`@test` under `tests/` (217 scenarios) with exit-0 and full pass.

Per-test median scenario cost is ~70 ms (measured via `bats --timing`);
the wall-clock deltas below are almost entirely bats + bash process-
spawn overhead, not test logic.

## 2026-04-22 — initial baseline (commit b02c791)

| Config           | Wall-clock | User CPU | Sys CPU | % CPU | Pass/Total | Speedup vs serial |
|------------------|-----------:|---------:|--------:|------:|:----------:|------------------:|
| serial           | **18:36**  |  221 s   |   20 s  |  21 % |  217 / 217 | 1.0× (baseline)   |
| `--jobs 2`       |  *skipped* |   —      |    —    |   —   |      —     | —                 |
| `--jobs 4`       | **1:27.7** |  168 s   |   26 s  | 221 % |  217 / 217 | **12.7×**         |
| `--jobs 8`       | **0:59.9** |  159 s   |   22 s  | 302 % |  217 / 217 | **18.6×**         |

### Interpretation

- Serial runs at 21 % CPU on a single core — 79 % of wall-clock is
  blocked on process creation + fsync, not actual work. Suite is
  embarrassingly parallel.
- `--jobs 4` captures ~93 % of the maximum achievable speedup on
  an 8-core box (1.008 s/test × 217 = algorithmic floor ≈ 60 s).
- `--jobs 8` adds only ~28 s of additional savings over `--jobs 4`
  while roughly doubling the CPU contention. Risk of flake under
  full-core saturation is higher.

### Chosen default

**`--jobs 4`** for `tests/run-all.sh`. Reasons:
- 93 % of the maximum speedup without saturating cores.
- Leaves headroom for concurrent work (editor, agent, other terminals).
- Lower risk of scheduler-induced flakes on the concurrency-
  deliberate bats scenarios (test_post_tool_use.bats concurrent
  dual-fire, state.bats concurrent load-modify-save).

Power users can override: `./tests/run-all.sh --jobs 8` forwards to bats.

### Prerequisite

GNU `parallel` must be on PATH for bats `--jobs` to function.
On Ubuntu/Debian: `sudo apt-get install -y parallel`. On macOS:
`brew install parallel`. The project's devcontainer includes it
as of 2026-04-22 (installed during Unit 1 of this plan).
Without `parallel`, bats silently runs 0 tests — the wrappers in
Unit 2 probe for it explicitly and exit 1 with a clear message.
